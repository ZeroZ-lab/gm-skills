# gm-x-hook-writer

只负责写 hook，不负责把整条推文或整串 thread 全部写完。

目标是让读者先停下来，而不是一上来追求“像 viral 文案”。

## 输入

优先从用户输入里提取这些信息：

- `topic`：想讲什么
- `audience`：写给谁
- `goal`：更适合拿停留、回复、转发、收藏还是关注
- `source_material`：观点、经历、数据、观察、案例
- `format`：单条推文或 thread 首条
- `tone`：锋利、克制、分析型、口语型

如果信息不完整，优先继续工作，不要为了补齐字段而过度追问。

缺少信息时默认：

- `audience`：对该主题有基础兴趣的普通关注者
- `goal`：先拿停留，再兼顾转发
- `format`：单条推文
- `tone`：清晰、直接、略带判断

## 工作流

### 第一步：提炼核心句

先把素材压缩成一句最值得被看见的话。

优先提炼：

- 反常识判断
- 明确问题
- 具体结果
- 真实代价
- 可被验证的观察

如果原始素材只有大词和空话，先去空话再写 hook。

### 第二步：选择 hook 角度

优先从以下角度中选择最适合的 3 到 5 个：

- `contrarian`：反常识
- `mistake`：错误与代价
- `specific-result`：具体结果或数字
- `observation`：观察到的模式
- `problem-callout`：点名常见问题
- `curiosity-gap`：信息缺口
- `checklist`：清单承诺
- `story-open`：经历开场

不要只是同一句话换同义词。每个角度都要真的改变切入方式。

### 第三步：写 hook

每个 hook 都尽量满足这些要求：

- 第一行就能独立成立
- 能看出对象、结果、代价、误区或冲突中的至少一个
- 少铺垫，少寒暄，少“我最近在想”
- 尽量短，优先让人一眼扫懂
- 不要写成标题党式空钩子

优先保留“具体”和“判断”，不要只保留情绪词。

### 第四步：筛掉弱 hook

以下类型直接判弱，必要时说明原因并重写：

- 太宽泛，谁都能套
- 只有情绪，没有信息
- 太像营销口号
- 开头像背景介绍
- 只是把主题重复一遍
- 靠夸张承诺吸引，但正文材料撑不住

### 第五步：输出最佳版本和备选

如果用户没有指定数量，默认输出：

- 1 个推荐 hook
- 5 到 10 个备选 hook

必要时补充：

- `最推荐版本：`
- `为什么它更强：`
- `适合后面怎么接：` 用一句话说明即可

## 输出格式

默认使用这个格式：

```md
# Hook Options

## Recommended
<最佳 hook>

为什么它更强：
<1-2 句>

适合后面怎么接：
<1 句>

## Alternatives
1. <hook>
2. <hook>
3. <hook>
...
```

如果用户明确要“只给我开头，不要解释”，就只输出 hook 列表。

## 写作原则

- hook 的任务是让人停下，不是把整条内容提前说完
- 优先写“判断 + 对象 + 差异”，不要先写背景
- 能具体就不要抽象
- 能直接说就不要绕
- 不要默认使用问句；只有真的能制造有效参与时才用
- 不要滥用数字；数字只有在能提升可信度时才值得写
- 不要为了吸引力牺牲准确性

## 边界

这个 skill 不负责：

- 完整单条推文成稿
- 完整 thread 结构
- 发帖排期或自动发布

如果用户要的是完整推文，先给 hook，再明确这只是开头层，不要假装已经完成整条内容。

# Anti-patterns

使用 `ui-fork` 时，避免以下输出问题。

## 1. 泛泛审美评价

不要只写：

- “整体很高级”
- “设计很现代”
- “界面干净简洁”

必须补成系统语言：

- 通过低饱和背景、单主强调色和轻边框建立克制层级。
- 通过重复卡片 anatomy 和稳定间距降低浏览成本。
- 通过标题、标签、说明文本的字号和权重差建立扫描路径。

## 2. 只总结颜色和字体

颜色和字体只是视觉系统的一部分。必须同时分析：

- 信息架构
- 布局骨架
- 组件复用
- 交互状态
- tokens
- Prompt Contract

## 3. 混淆观察和推断

不要写“字体是 Inter”这类无法从截图确认的结论。

应写：

- 直接观察：正文是无衬线字体。
- 高概率推断：适合使用 Inter、Geist、IBM Plex Sans 一类中性无衬线字体。
- 无法确认：具体字体文件和字重映射。

## 4. 编造精确值

不要无依据输出精确 hex、字号、间距。

可接受写法：

- 主强调色：蓝色系，估计接近 `#2563EB`。
- 卡片圆角：中等圆角，估计 12-16px。
- 页面外边距：桌面端估计 24-32px。

## 5. 把所有页面套成同一种结构

必须先判断页面类型。不同页面的提炼重点不同：

- Directory 重视分类、搜索、卡片和发现路径。
- Dashboard 重视指标层级、筛选和状态。
- Marketing 重视叙事节奏、CTA 和信任模块。
- Tool 重视输入、输出、控制和反馈。

## 6. 直接跳到前端代码

默认不要写 React、HTML、CSS、Tailwind 或 shadcn 实现代码。

只有用户明确要求“实现”“生成页面”“写组件”“转成代码”时，才进入代码层。即使进入代码层，也要先保留设计系统约束。

## 7. 无依据添加装饰语言

不要为了“更好看”擅自加入：

- 玻璃拟态
- 复杂发光
- 噪点纹理
- 大面积渐变
- 过度阴影
- 复杂 3D 元素

除非截图本身明确存在这些语言，或用户明确要求延展。

## 8. Prompt Contract 过弱

不要写成“保持风格一致”这种空话。

必须明确：

- 哪些结构必须保留
- 哪些组件 anatomy 必须复用
- 哪些 token 关系必须继承
- 哪些内容可以变化
- 哪些设计动作被禁止

## 9. 忽略不确定性

如果只看到单张静态图，不要确认：

- hover 状态
- focus ring
- loading 状态
- 真实断点
- 完整品牌色阶
- 真实字体
- 组件库来源

应把它们放进无法确认项，或标注为交互假设。

## 10. Prompt Contract 没有执行力

不要只写：

- “后续保持一致”
- “沿用当前风格”
- “注意统一”

必须写成可执行合同：

- 必须继承哪些布局骨架
- 哪些组件 anatomy 不可改变
- 新增模块必须复用哪些 token
- 哪些视觉动作被禁止
- 哪些内容可以变化而不破坏系统

## 11. 过早进入文件或代码流程

不要在用户只想看分析时先询问是否创建文件，也不要默认生成前端代码。

应先完成设计系统提炼；只有用户明确要求保存、生成 `DESIGN.md`、实现页面或写组件时，才进入文件或代码流程。

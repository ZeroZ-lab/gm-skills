# {{产品/页面名称}}设计指南

**版本**：v1.0
**适用范围**：说明该规范适用于哪些页面、产品或场景。
**样本来源**：说明截图数量、主题版本、页面状态或用户提供的背景。
**文档目标**：为后续 AI 生成页面、设计扩展、组件复用、前端落地提供统一约束。

> 使用规则：这是固定输出骨架。除非用户明确要求简化，否则不要省略关键资产；如果某一项不适用，写明“不适用”或“截图无法确认”，不要虚构内容补齐。最终产物必须形成四段式文档包：Design Brief、Design Guide、Design Tokens Draft、AI Prompt Contract。

## 1. Design Brief / 设计定位

### 1.1 产品类型

说明页面属于 Marketing、Dashboard、Workspace、Directory、Content、Tool、Form、Mobile App、AI Workbench 或混合类型。必须区分主类型和次类型。

### 1.2 页面目标与产品任务

说明页面服务的产品目标或业务目标，不只描述画面内容。

### 1.3 用户主任务

说明用户进入该页面最主要要完成什么。

### 1.4 主阅读路径与辅助路径

说明用户视线和操作路径如何展开。

### 1.5 页面气质关键词

只写能被结构、色彩、排版或组件规则支撑的关键词。

### 1.6 适用范围与不适用范围

说明该设计系统适合继续生成哪些页面，不适合套用到哪些页面。

## 2. Design Principles / 设计总原则

提炼该界面背后的设计原则。每条原则必须解释它如何影响布局、组件、视觉或交互。

建议覆盖：

- 信息优先还是品牌优先
- 模块化程度
- 主题切换关系
- 信息密度策略
- 首页或当前页在产品中的角色
- 后续 AI 生成必须受哪些结构约束

## 3. Information Architecture / 信息架构

### 3.1 页面一级结构

列出主要区域和顺序。

### 3.2 主次区关系

说明主内容区、辅助区、导航区、行动区之间的权重关系。

### 3.3 阅读路径

用箭头或顺序列表说明用户如何从入口、主任务、辅助发现、行动路径中移动。

### 3.4 信息分组逻辑

说明内容为什么这样分组，哪些模块承担认知、浏览、筛选、背书或转化。

## 4. Layout System / 布局系统

### 4.1 总体布局模式

描述单列、双列、多面板、顶部导航、底部导航、卡片流、详情编辑等布局范式。

### 4.2 容器策略

说明页面主容器、模块容器、内容单元容器的关系。

### 4.3 栅格推断

给出栅格、列宽、断点和响应式推断；无法确认时标注为估计。

### 4.4 间距系统

说明页面级、模块级、组件级间距节奏。可给出 4/8/12/16/24/32/40/48 等建议，但必须标注为建议或估计。

### 4.5 可扩展结构规则

说明新增模块、删减模块、扩展页面时应如何保持结构连续性。

## 5. Visual System / 视觉语言

### 5.1 主题结构

说明该系统应拆成基础层 token 与主题层 token 的哪些部分。

### 5.2 色彩体系

说明背景、表面、文字、强调色、状态色、边框的层级关系。

### 5.3 明暗主题关系

如截图包含明暗主题，说明哪些是主题变量，哪些布局规则保持不变。

### 5.4 层级建立方式

说明层级主要由背景对比、边框、阴影、留白、字体层级还是色彩强调建立。

### 5.5 圆角、边框与阴影系统

说明不同组件的圆角层级和一致性规则。

### 5.6 视觉边界

说明哪些视觉动作会破坏原系统，例如过度渐变、过强阴影、无依据装饰或过多强调色。

## 6. Typography / 字体与排版规范

### 6.1 排版风格判断

说明它是平台型 UI 排版、营销型叙事排版、数据型排版、编辑型阅读排版，还是工具型任务排版。

### 6.2 字体层级

说明 H1、H2、H3、Body、Caption、Meta、数字文本或代码文本的层级关系。

### 6.3 排版规则

说明标题长度、描述行数、截断规则、数字突出方式、meta 弱化方式和可读性约束。

## 7. Component System / 组件系统

对每个重要组件输出以下字段：

- 作用：
- 结构：
- 层级：
- 复用规则：
- 可变项：
- 不可变项：
- 状态与交互：
- 相关 tokens：

优先分析截图中真实存在的组件。常见组件包括：

- 导航栏
- Hero
- 搜索
- 按钮
- 标签/分类
- 卡片
- 列表/趋势/数据模块
- 侧栏
- 图表
- 分页
- 页脚

## 8. Interaction Guidelines / 交互规范

### 8.1 交互基调

说明交互应偏快速浏览、精确操作、沉浸阅读还是任务完成。

### 8.2 Hover / Focus / Active

说明可推断的状态变化规则。

### 8.3 点击反馈与状态切换

说明筛选、导航、标签、主题等切换规则。

### 8.4 动效建议

说明动效节奏、持续时间、使用边界。

### 8.5 主题切换规则

如适用，说明主题切换时布局、组件和 token 的继承关系。

## 9. Design Tokens Draft / 设计变量草案

说明：tokens 是草案，不是最终权威值。无法确认的值留空或标注“未知”；估计值必须标注“估计”。

### 9.1 Color Tokens

使用 `tokens_template.yaml` 中的 color 结构。

### 9.2 Typography Tokens

使用 `tokens_template.yaml` 中的 typography 结构。

### 9.3 Spacing Tokens

使用 `tokens_template.yaml` 中的 spacing 结构。

### 9.4 Radius Tokens

使用 `tokens_template.yaml` 中的 radius 结构。

### 9.5 Border Tokens

使用 `tokens_template.yaml` 中的 border 结构。

### 9.6 Shadow Tokens

使用 `tokens_template.yaml` 中的 shadow 结构。

### 9.7 Motion Tokens

使用 `tokens_template.yaml` 中的 motion 结构。

### 9.8 Semantic Tokens 与 Component Tokens

说明哪些 token 应作为语义变量暴露给前端或后续 AI，例如 `bg.default`、`text.primary`、`action.primary`、`card.border`。

## 10. Frontend Handoff Notes / 前端落地建议

当用户要求 `system` 模式或前端落地时输出。建议拆成：

- Design Tokens
- Semantic Tokens
- Component Spec
- Page Template
- Theme Strategy
- Responsive Rules

不要直接写完整前端代码，除非用户明确要求实现。

## 11. AI Prompt Contract / 给 AI 的生成约束

### 11.1 后续 AI 设计必须遵守的规则

列出必须继承的页面类型、结构、组件、视觉和交互规则。

### 11.2 可以变化的部分

列出可替换内容、可扩展模块和可局部调整的视觉参数。

### 11.3 禁止破坏的骨架

列出不允许改变的布局范式、组件 anatomy、主题关系和信息架构。

### 11.4 新增组件扩展规则

说明新增组件必须如何继承 spacing、radius、border、typography、state 和 motion 规则。

### 11.5 禁止项

列出后续 AI 不得擅自引入的视觉语言、布局范式和交互模式。

## 12. Extraction Checklist / 关键提炼项

列出本次已经提炼出的产品层、结构层、视觉层、组件层、交互层、实现层资产。

## 13. Core Design Conclusion / 核心设计结论

用一段压缩判断说明该截图体现的不是“某种风格”，而是什么可复用的设计系统原型。

## 14. Confidence Split / 观察、推断与未知

### 14.1 直接观察到的事实

只列截图中明确可见的内容。

### 14.2 高概率推断

列出基于截图和经验推断出的结论。

### 14.3 无法确认项

列出需要更多页面、交互状态或品牌资料才能确认的内容。
